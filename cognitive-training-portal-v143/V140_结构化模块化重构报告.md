### 1.1 代码规模

### 1.2 当前架构问题

**问题1：单文件巨石架构**
- 所有代码（HTML + CSS + JS + 数据）挤在一个 index.html 中
- 任何修改都可能在其他位置引发连锁问题
- 历史记录：week5Plan被截断过3次、HTML模板混入script内部2次、逗号缺失导致全站崩溃1次

**问题2：全局变量污染**
- 103个全局变量无命名空间隔离
- 变量命名冲突风险高（如 topicsData、week5Plan 等可能被意外覆盖）
- 模块间隐式依赖，无法追踪谁使用了哪些变量

**问题3：数据与逻辑混合**
- 母题数据（378题）约2000行和渲染逻辑在同一文件
- 播客数据（21条）约200行散落在JS中间
- Week1-7训练计划数据约600行，曾3次被其他代码覆盖截断

**问题4：函数无边界**
- 400+函数在同一全局作用域，互相调用关系不明
- 修改一个函数可能影响其他模块的同名变量
- "其他"分类下有143个函数无法归属到具体模块

**问题5：HTML模板内嵌**
- 所有UI通过 innerHTML 字符串拼接生成
- HTML模板代码散落在JS函数内部
- 调试困难，语法错误难以定位

### 1.3 历史故障统计

| 故障类型 | 发生次数 | 根因 |
|----------|----------|------|
| JS语法崩溃（全站白屏） | 5次 | sed插入代码位置错误/覆盖相邻变量 |
| 重复函数定义 | 3次 | 多次升级注入同名函数 |
| 数据截断（week5Plan/week6Plan） | 3次 | 新代码覆盖了相邻数据定义 |
| 播放器状态残留 | 2次 | 模块切换未清理上一模块状态 |
| 逗号缺失 | 1次 | 数组/对象新增条目后缺少分隔符 |

**核心结论**：90%的故障源于"修改A模块时意外破坏了B模块"，这是单文件无隔离架构的必然结果。

### 2.2 模块隔离原则

**原则1：文件即边界**
- 每个模块一个文件，修改只影响该文件
- 模块间通过 CTM 钩子通信，不直接调用内部函数

**原则2：数据与逻辑分离**
- 数据文件（js/data/）只负责声明和导出数据
- 逻辑文件（js/modules/）只负责渲染和交互
- 数据文件修改不会影响逻辑，反之亦然

**原则3：统一出口**
- 每个模块通过 CTM.registerModule() 注册
- 渲染函数统一挂载到 window（供HTML onclick调用）
- 模块内部变量不暴露到全局

**原则4：向后兼容**
- HTML中的 onclick="xxx()" 继续工作
- 现有 localStorage 数据格式不变
- URL路由和模块切换逻辑不变

## 六、风险与应对

| 风险 | 应对策略 |
|------|----------|
| GitHub Pages不支持ES Module | 使用传统`<script>`标签加载，不使用import/export |
| 扣子平台文件同步 | 所有文件推送到GitHub仓库，扣子平台自动同步 |
| 首次加载变慢（多文件请求） | 文件总数可控（~20个），浏览器并发加载；后续可合并压缩 |
| onclick全局函数失效 | 模块加载后将渲染函数挂载到window，保持向后兼容 |
| localStorage数据兼容 | storage.js保持原有数据格式不变 |

---

**结论**：当前单文件架构已到极限，每次修改都像拆弹。模块化重构不是"锦上添花"，而是"必须做"。重构后，修改任何一个模块都不会影响其他模块，这才是结构化和模块化真正的意义。
